      This is NodeJS

      Post on Sep 4 2014

      今天翻之前的目标记录，有一条”Write an article about NodeJS”被放在第一行，现在来完成它。

      关于NodeJS，我认为下面几个东西比较有意思：

      1.异步和回调；
2.实时通信;
3.模块化机制

      异步和回调

      前端工程师对这两个概念一定不会陌生，但是很多从Java或者C#转过来的JSer经常在这里出问题。由于对异步理解的薄弱，在使用最基础的ajax时写法粗糙，最常见的低级错误：

      // 请求
function getData() {
    $.ajax({
        url: '/data',
        success: function(data) {
            return data;
        }
    });
}

// 调用
var myData = getData();

      调用的时候myData始终是undefined，这个现象没什么好解释的，正确的做法应该是这样：

      // 请求
function getData(callback) {
    $.ajax({
        url: '/data',
        success: function(data) {
            callback(data);
        }
    });
}

// 调用
getData(function(data) {
    // 对取到的数据进行操作...
});

      这种写法在前端分离数据访问层的时候非常有用，可以把所有向WebApi的数据请求单独维护。在jQuery 1.5 版本引入deferred对象之后，可以用另一种链式写法类似：

      // 请求
function getData() {
    return $.ajax(url);
}

getData()
.done(//成功处理)
.fail(//失败处理)

      这块内容如果要说需要单独篇幅来介绍，暂时不展开，想了解的可以查看官方API:http://api.jquery.com/jQuery.Deferred/。

      在NodeJS中，前一种回调函数的方式是最为普遍的，其中有一个重要约定：

      回调函数第一个参数一定是error信息;

      这样，当我们向上一层传递数据的时候，出错情况下：

      callback(err, null);

      正常情况下：

      callback(null, data);

      在数据操作层，可以通过err对象对操作状态做出判断并对异常做处理。

      但是，一切皆回调的方式很容易让回调层级变得越来越深，在完全习惯之前代码可读性会有很大影响。还好，有很多工具可以帮我们解决这个问题，这里介绍一个朴灵写的EventProxy，名称已经足够清楚，通过事件代理避免深度嵌套问题。

      有以下场景，我想组一个球队踢比赛，两个步骤是必须的：

      1.找一名教练；
2.组一批球员；
3.开始比赛

      按照异步逻辑处理，这种问题是比较头疼的，步骤一与步骤二之间互相独立，没有因果关系，但是步骤三却需要一、二都完成之后才能触发。

      大家可以尝试用自己的方式实现它，这里用EventProxy的处理代码是这样：

      var ep = EventProxy.create('cotch', 'players', function (cotch, players){
    // 赢球!
});

$.get("cotch", function (cotch) {
    ep.emit("cotch", cotch);
});


$.get("players", function (players) {
    ep.emit("players", players);
});

      这只是最基础的场景，其中第一句是由两步操作组成：

      1.实例化ep对象；
2.注册所有事件完成后的回调事件

      这样的写法，依赖关系、代码逻辑都十分清晰。同时，EventProxy也支持部分高级用法，在前端博客中实现无限层级评论(就是给评论回复，给回复再回复，PS:没有上线)就是使用重复异步协作配合递归函数实现，在所有评论异步获取完成之后再执行操作。总体来说，EventProxy使用起来非常简单，学习成本极低，不管是客户端JS还是服务端JS，都值得了解。

      实时通信

      提到实时通信，必须要说WebSocket。原本WebSocket是不应该单独归于NodeJS范围的，但是由于JavaScript的特性，导致在Node中实时通信变得异常简单，比想象的更简单。所以顺便在这做些介绍。

      Node中有单纯实现WebSocket协议的库，但是更加常用的是另外一个socket-io，类似jQuery的兼容性处理，由于socket-io同时支持包括WebSocket在内的多种协议，在不同运行环境下，它会自动使用最适合的通信方式。当我们用jQuery绑定事件的时候，可以不关心是否是IE平台，是否支持标准浏览器的addEventListen这些兼容性问题，一个on方法解决。socket-io也是一样，我们只需要把精力放在设计和使用上，这是我喜欢的。

      下面是一个上传进度通信的Demo，在服务端，核心代码如下(需要socket.io库)：

      var progress = 0, si;
io.on('connection', function (socket) {
    // 接收启动信号
    socket.on('start', function() {
        console.log('start');
        // 发出进度信号
        // 模拟每秒进度+1
        si = setInterval(function() {
            socket.emit('update', {
                progress: progress++
            });
        }, 1000);
    });

    // 接收结束信号
    socket.on('stop', function () {
        clearInterval(si);
        progress = 0;
    });
});

      客户端类库在这里：socket.io-1.0.6.js，引入这个类库之后，使用jQuery的JS代码如下：

      $(function() {
  var socket = io();
  $("#start").click(function() {
    socket.emit('start');
  });

  $("#end").click(function() {
    socket.emit('stop');
  });

  socket.on('update', function(data) {
    $("#progress").text(data.progress);
    if(data.progress >= 100) {
      socket.emit('stop');
      alert("上传成功了");
    }
  });
});

      页面有一个开始、停止按钮和一个进度显示区域。这就是全部。

      随着项目产品化程度越来越高，实时通信的需求一定只增不减，之前的项目中遇到类似进度处理(这个需求非常非常普遍)，都是使用ajax轮询，但是我认为，不管是从交互体验还是实现上，都不应该是这样，数据应该更灵活更真实。

      模块化机制

      Node的模块遵循CMD规范，和Seajs类似。

      从根本上说，封装机制就是两部分：

      1.引入；
2.暴露

      在Node的模块化中，以下几个关键字是最常见的：require和module.exports。

      其中require是引入模块的关键字:

      var express = require('express')

      module.exports就是当前模块向外暴露的接口，比如在file1.js中：

      module.exports = 1;

      那么在file2.js中：

      var param = require('./file1.js');

      这个时候的param就是file1中暴露出的对象。

      在Node中，有时我们使用module.exports，有时也会直接使用exports。关于exports和module.exports的区别，基本属于JS语法的内容，但是在SO里面有这么一个问题，有几个答案很不错。排名第一的答案解释很具体，但是我认为排名第二的答案更有意思：http://stackoverflow.com/questions/7137397/module-exports-vs-exports-in-nodejs。

      这个知识点很重要，放出我写的demo帮助理解：

      // 放到浏览器控制台运行
var module = {};
module.exports = 13;
(function(exports, module){
    exports = 0; // 然后把这行换成exports = module.exports = 0;试试
    console.log(module.exports);
})(module.exports, module);

      所以在设计每一个文件(模块)的时候(不仅限于此)，最先思考的应该是：需要哪些资源 和 暴露哪些接口，简单理解就是 需要什么 和 输出什么。所有模块的设计核心都应该基于这个理念，哪怕单个模块内部实现是混乱的，但是暴露的接口一定保证清晰可用。

      基本知道这些就可以去实践了，因为遵循相同的规范，习惯了Node的模块化机制之后，再去学习Seajs可以很快上手。最后，关于RequireJS和SeaJS，虽然规范不同，但基本上每一个都能满足项目需求。RequireJS案例更多，出问题更好解决，所以个人更加推荐。
